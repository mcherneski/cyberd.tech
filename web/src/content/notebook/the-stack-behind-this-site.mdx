A personal site is a small system, which makes it a fair test of judgment: every component you add is a component you maintain, patch, and pay for. This site is the architecture I recommend when someone asks how to publish on the modern web without acquiring a second job. Here is the full stack, and the reasoning behind each layer.

## The principle: static first, dynamic narrowly

Most of what a portfolio does — pages, articles, project cards, structured data — is known at build time. Rendering it per-request buys nothing except servers to secure and bills to audit. So the foundation is a static site, and the dynamic surface is exactly one path: the contact form. Everything else ships as files.

That single decision drives most of what follows. Compliance surface shrinks. Cost approaches zero. There is no database because nothing here needs one.

## The build: Astro, Tailwind, MDX

**Astro** generates the site. It renders to plain HTML by default and ships JavaScript only where a page opts in — here, that is a project filter and a theme toggle, measured in kilobytes. Content lives in **MDX and JSON collections** validated by schema at build time: a project card with a missing field fails the build, not the reader.

**Tailwind CSS** handles styling against a small set of design tokens (color, shadow, type). The visual system — the brutalist-adjacent borders, the accent palette, the softened shadows — is a few hundred lines of CSS on top of utilities, which keeps restyling cheap.

**Pagefind** indexes the built HTML and serves search from static files. No search service, no API key, no per-query cost.

## The delivery: S3, CloudFront, ACM, Route 53

The built site lands in a **private S3 bucket**. Nothing is publicly readable; **CloudFront** fetches objects through **Origin Access Control**, so the CDN is the only door. A CloudFront function rewrites clean URLs to their `index.html` objects. **ACM** issues the TLS certificate, and **Route 53** aliases `cyberd.tech` to the distribution.

This is the boring, durable pattern: global caching, TLS by default, an origin that cannot be scraped or hotlinked around, and a monthly bill that rounds to pocket change.

## The dynamic layer: one Lambda, deliberately

The contact form posts to **API Gateway** backed by a single **Lambda** function, which validates the submission, checks a **Cloudflare Turnstile** token so bots do the failing instead of humans, and forwards the message through **SES**. No submission is stored by default — the function is a relay, not a database. That is a privacy posture, not an accident: data you do not retain is data you cannot leak.

## The infrastructure: CDK, so the whole thing is code

Every resource above — bucket, distribution, certificate, DNS records, function, API — is defined in an **AWS CDK** stack in the same repository as the content. Deploys are `git push`: GitHub Actions builds the site, synthesizes the stack, and ships both. The infrastructure is reviewable in a pull request and reproducible from a clean checkout, which is the same discipline I hold client systems to. A system you cannot rebuild from source is a system you do not fully own.

## The AI-native layer: two surfaces, one source

The part most sites skip: this site publishes for two kinds of readers. Humans get the HTML. Agents get a parallel surface generated from the same content at build time — **llms.txt** and **llms-full.txt** indexes, a **Markdown report for every page** (`/projects/moxa.md` and so on), and **JSON-LD** structured data describing the person, the projects, and the writing.

None of this costs anything at runtime; it is all static output. But it means an AI system researching me on someone's behalf gets curated, canonical context instead of scraping a nav bar. As more buying research is delegated to agents, I consider this the difference between being findable and being legible.

## What it costs to run

- **Compute:** pay-per-request Lambda that spends most of its life at zero.
- **Storage and delivery:** S3 and CloudFront at personal-site traffic, effectively rounding error.
- **Maintenance:** dependency updates. There is no server to patch and no database to back up.

## The takeaway

The stack is not impressive because any single piece is exotic — it is not. It is a set of boring components, each chosen because it can be fixed at 2am, composed so the whole thing runs itself. That is the standard I hold for enterprise systems, applied at the smallest possible scale. If you are building a site like this one, steal the pattern: static first, one narrow dynamic path, infrastructure as code, and a machine-readable surface next to the human one.
