# Founder Operating System

## One-line pitch

A minimal operating layer for founders: enough structure to move fast with evidence, not enough process to become a second job.

## The problem

Founders oscillate between two failure modes:

- **No system** — Ideas, customer notes, and commitments live in scattered threads; nothing compounds.
- **Too much system** — Enterprise PM theater (OKR decks, status meetings) slows a team of one to five to a crawl.

## The approach

A small set of **primitives** you can adapt weekly:

### Bets

A bet is a hypothesis with a time box: *We believe X for audience Y; we’ll know by date Z.*

Each bet has:

- Expected signal (qualitative or quantitative)
- Maximum investment (time / money)
- Kill criteria (what would make us stop)

### Evidence

Customer conversations, usage metrics, and shipped artifacts attach to bets—not to a generic “backlog.” Evidence answers: *Did this bet get stronger or weaker?*

### Constraints

Explicit limits: runway, team capacity, compliance boundaries, technical debt budget. Constraints prevent “yes” by default.

### Next actions

One visible next action per active bet. If everything is priority, nothing is.

### Review cadence

Short weekly review: which bets advance, which stall, which die. No multi-hour planning marathons.

## Artifacts for collaborators and agents

- **Project briefs** — One page: problem, bet, evidence so far, open questions, next action.
- **Agent-readable summaries** — Markdown siblings (like this site’s `.md` reports) so tools can reason about context without re-reading Slack.

## Who this is for

- Technical founders shipping with a tiny team
- Fractional CTO / advisor engagements needing shared clarity
- Pre-PMF products where speed and learning rate dominate
